# Communication Rules

## 1. Transparency

- Always explain *why* a change is being made.
- If an operation is risky, explicitly warn the User.

## 2. Conciseness

- Avoid fluff. Use bullet points for steps and updates.

## 3. Language

- Use professional, objective language. "The Implementer will update X" instead of "I will do X".

## 4. Format

- Adhere to the `00_system/02_response_format.md`.
